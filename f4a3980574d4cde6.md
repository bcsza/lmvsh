# Documentation Bridge

Documentation Bridge 是一个面向开发者与技术文献工作者的开源外链资源聚合与结构化导航系统。该项目不托管任何原始内容，而是通过人工筛选与社区贡献，建立高质量技术文档、工具链站点、学术资源与行业白皮书的索引网络。项目定位为“技术文档的文档”，即帮助用户在海量信息中快速定位权威、稳定、可追溯的在线资源，特别适用于技术选型调研、系统设计参考、离线文档规划与团队知识库初始化等场景。

目标用户包括基础架构工程师、技术文档撰写者、开源项目维护者以及计算机科学领域的研究人员。项目通过严格的链接可用性检测、内容分类标签与版本快照引用，显著降低用户在多个技术社区间反复跳转的认知负担，同时提供可插拔的元数据扩展接口，便于企业内部分支定制。

## 功能概览

- **多维度资源分类**：按技术领域、资料类型、维护状态与活跃度对每条外链添加系统标签，支持自由组合过滤。

- **自动化可用性探针**：每日定时检测所有收录链接的 HTTP 状态码与响应时间，自动标记失效或降级节点，并提供近七天的可用率趋势数据。

- **结构化元数据提取**：对支持 Open Graph 或 JSON-LD 的页面，自动抓取标题、描述、作者信息与更新日期，补充至索引库。

- **社区贡献工作流**：基于拉取请求的链接增删改流程，每个提交需附带链接用途说明、分类建议与至少一个备选镜像地址。

- **快照引用生成器**：为每条外链生成稳定的引用标识符，支持 Markdown、reStructuredText 与 AsciiDoc 格式的引用片段复制。

- **离线阅读包导出**：允许用户按标签筛选后，生成包含链接清单与摘要信息的静态 HTML 或 JSON 文件，便于内网分发或文档归档。

- **变更历史追溯**：记录每条链接的添加时间、修改记录与状态变更日志，所有操作附加贡献者信息与提交哈希。

## 应用场景

- **技术选型调研**：团队在引入新中间件或框架时，可通过本项目快速获取官方文档、社区最佳实践、性能测试报告与安全公告的聚合入口，减少多搜索引擎重复检索的时间成本。

- **离线文档体系建设**：企业内网环境受限时，文档管理员可利用本项目的导出功能，批量获取外链摘要与元数据，辅助制定内部镜像同步策略或下载优先级清单。

- **开源项目 README 外链维护**：开源项目维护者可将本项目作为子模块引用，避免在主仓库中维护大量易失效的外部链接，同时借助本项目的可用性检测机制主动感知链接变更。

- **技术培训课程资料整合**：培训机构或高校讲师可按课程模块筛选相关链接，生成定制化的课外阅读材料列表，并利用快照引用功能确保教材中的链接长期可追溯。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户建议通过 WSL2 或 Git Bash 执行。

```bash
# 克隆仓库至本地
git clone https://github.com/doc-bridge/doc-bridge.git
cd doc-bridge

# 安装项目依赖（使用 Python 3.10 及以上版本）
pip install -r requirements.txt

# 初始化本地索引数据库并执行首次全量同步
python manage.py init-db
python manage.py sync --full

# 启动开发服务器，默认监听 127.0.0.1:8080
python manage.py runserver
```

完成上述步骤后，访问 `http://127.0.0.1:8080` 即可查看本地索引面板。首次同步可能耗时数分钟，具体取决于网络环境与收录链接总数。

## 贡献指南

1.  **分支与提交规范**：从 `main` 分支创建新的特性分支，分支命名遵循 `feat/`、`fix/`、`docs/` 前缀。提交信息需使用英文，采用 `type(scope): subject` 格式，正文简要说明动机与影响。

2.  **链接添加流程**：在 `data/links.yaml` 中按分类添加新条目，必须包含 `url`、`title`、`category` 与 `notes` 字段。提交拉取请求前，本地执行 `make check` 验证 YAML 语法与链接可达性。

3.  **测试覆盖率要求**：所有新增或修改的 Python 代码需附带对应的单元测试，核心模块的测试覆盖率不得低于 85%。运行 `make test` 可查看当前覆盖率报告。

4.  **文档同步更新**：若修改涉及用户可见功能，需同步更新 `docs/` 下对应的章节，并在拉取请求描述中标记 `[docs]` 标签。

5.  **审阅等待周期**：拉取请求提交后，至少等待 48 小时供其他贡献者审阅。审阅意见需逐条回复或采纳，最终由至少一名核心维护者批准合并。

## 常见问题

**Q：收录链接失效后，系统会如何处理？**

A：可用性探测程序每六小时执行一次全量检测。首次发现失效时，系统在管理面板标记为“预警”状态并发送邮件通知最近更新者。若连续三次检测均不可达，则自动标记为“已失效”，该链接仍保留在索引中但默认不展示于公共视图。贡献者可提交拉取请求更新链接地址或移除条目。

**Q：是否可以添加非技术类或商业性质的链接？**

A：本项目的核心定位为技术文档与开发资源导航，原则上不接受纯商业广告、电商页面或与软件开发无关的站点。但具有高参考价值的行业白皮书、标准规范文档或学术论文页面，经社区讨论后可以收录。提交时请在 `notes` 字段中明确说明链接的技术相关性。

**Q：本地索引数据库如何迁移至生产服务器？**

A：项目原生支持 SQLite 文件迁移，直接复制 `data/index.db` 至目标服务器即可。若使用 PostgreSQL 或 MySQL，需通过 `config/settings.py` 修改连接字符串，并执行 `python manage.py migrate` 初始化表结构。随后运行 `python manage.py sync --from-db` 从原库导入数据，该操作支持断点续传。
