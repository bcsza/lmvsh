# OpenResourceHub

OpenResourceHub 是一个面向开发人员、技术研究者与内容创作者的轻量级技术资源导航与外部链接聚合平台。该项目不提供具体的软件工具或托管服务，而是以人工筛选与社区共建的方式，整理互联网中高质量、低冗余的技术文档、实验环境、媒体素材与数据源站点，帮助用户快速定位可靠的外部资源，减少信息检索成本。

项目定位为“技术型外链汇总与质量评估工具”，适用于需要频繁查找外部技术参考、测试环境、视觉素材或实验数据的用户群体。OpenResourceHub 通过结构化的分类索引、可用性检测与访问速度标记，使得用户无需反复验证站点可用性，即可直接使用经过初步筛选的链接资源。

---

## 功能概览

- **资源分类索引** 按技术领域、资源类型与访问用途对收录链接进行多维度标签分类，支持按名称、标签与描述进行过滤查询。

- **可用性健康检查** 对收录的外部链接进行定时可用性探测，自动标记无法访问、响应超时或证书过期的站点，并在前端界面中展示状态标识。

- **访问速度标记** 基于历史探测数据为每个链接标注相对访问速度等级（快/中/慢），辅助用户在高延迟网络环境下选择合适的资源入口。

- **社区提交与审核** 提供标准化的资源提交模板，允许社区用户推荐新链接，并经过 maintainer 审核后合并入主索引库。

- **结构化元数据导出** 支持将全部索引数据导出为 JSON、YAML 或 Markdown 表格格式，便于集成至其他文档系统或自动化工具链。

- **私有化部署支持** 项目基于纯静态前端与轻量后端 API 设计，可完整运行于内网环境，满足企业或实验室内部资源聚合需求。

- **变更历史追踪** 所有新增、删除或更新操作均记录时间戳与操作人，支持回滚与审计。

## 应用场景

- **技术文档编写与维护** 技术作者在编写教程、API 文档或操作手册时，可通过 OpenResourceHub 快速获取已验证的外部参考链接，避免因引用失效链接导致的文档质量问题。

- **CI/CD 流水线环境配置** 运维工程师在编写 Dockerfile 或自动化部署脚本时，需要引用外部软件源、镜像仓库或配置模板地址，可通过本项目的导出功能批量获取稳定的资源入口。

- **数据科学与可视化实验** 数据研究人员在准备演示环境或教学案例时，需要引用公开的数据集示例、视觉素材或演示视频源，可通过本项目的媒体资源分类快速定位可用站点。

- **网络安全与渗透测试练习** 安全研究人员在进行合法授权的测试练习时，可通过本项目的实验环境分类获取公开的靶场或沙箱入口，无需自行搭建基础环境。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，默认使用 Python 3.9+ 与 Node.js 18+。

```bash
# 克隆项目仓库
git clone https://github.com/your-org/OpenResourceHub.git
cd OpenResourceHub

# 安装后端依赖（使用 pip）
pip install -r requirements/backend.txt

# 安装前端依赖（使用 npm）
cd frontend
npm install
cd ..

# 启动本地开发服务（后端端口 8000，前端端口 3000）
python scripts/dev_server.py --port 8000 --frontend-port 3000
```

访问 `http://localhost:3000` 即可查看资源索引主页。默认加载 `data/seed_links.json` 中的初始链接集合。

## 贡献指南

1. 在 GitHub 上 fork 本项目仓库，并克隆到本地开发环境。确保本地安装所有必需依赖并通过 `make test` 验证现有测试全部通过。

2. 提交新资源链接时，请使用 `data/seed_links.json` 中定义的 JSON 格式，确保提供完整字段（标题、URL、分类、标签、描述），并将新增条目置于 `pending/` 审核分区中。

3. 对于现有链接的更新（如 URL 变更、描述修正），请直接修改对应条目并在 commit message 中注明 `update:` 前缀以及变更原因。

4. 提交 Pull Request 前，请运行 `make format` 进行代码格式化，并运行 `make lint` 进行静态检查。PR 描述中需列出新增或变更的链接数量与分类分布。

5. PR 合并前至少需要一位 maintainer 进行审核，审核内容包括链接可用性验证、内容合规性检查以及描述准确性评估。合并后 CI 将自动触发数据库重新构建与部署。

## 常见问题

**Q: 健康检查模块如何判断一个站点是否可用？**

默认采用 HTTP/HTTPS 的 TCP 连接握手结合 HTTP 状态码（200-399 视为成功）作为主要判断依据。对于非 HTTP 协议（如 FTP、SSH），则仅检测端口是否开放。检测超时时间设置为 5 秒，每个链接每次探测会进行 2 次重试。所有探测结果会记录响应时间与状态码，并用于更新速度标记。

**Q: 如何导入大量的初始链接？是否支持批量导入？**

项目提供了 `scripts/import_bulk.py` 脚本，支持从 CSV、JSON Lines 以及标准 Markdown 列表格式批量导入。导入前会自动进行去重校验（按 URL 完全匹配），并跳过已存在的条目。建议在导入前使用 `--dry-run` 参数进行预检，确认无误后再执行实际写入。

**Q: 生产环境下如何配置探测频率与并发数？**

探测相关参数通过 `backend/config/production.yaml` 文件进行配置。`check_interval` 字段控制全局探测周期（默认 3600 秒），`concurrency` 字段控制同时探测的链接数量（默认 20）。若部署于低性能服务器，建议将并发数调低至 5 以下，以避免资源耗尽。
