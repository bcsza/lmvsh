# Vanguard Link Aggregator

Vanguard Link Aggregator 是一个面向技术内容创作者与数字资源管理者的高可靠性外链聚合与导航系统。本项目并非传统意义上的通用书签管理工具，而是针对高频访问、多源异构的技术参考资料站点进行结构化组织与分类展示的静态门户生成框架。其核心定位在于为开发者社区、技术文档团队以及个人知识库维护者提供一套轻量化、可定制、低维护成本的资源索引方案，解决因资源分散、链接失效、分类混乱导致的信息检索效率低下问题。

本项目不依赖动态后端服务，完全基于静态 Markdown 与 YAML 数据源构建，支持一键生成完整的 HTML 导航页面，适用于部署在任意静态托管服务或本地文件系统中。通过预设的分类模板与自动化校验脚本，Vanguard Link Aggregator 能够帮助用户在短时间内建立清晰、可扩展的资源导航体系，并有效降低链接维护的认知负担。

## 功能概览

- **多级分类索引**：支持按技术领域、资源类型、使用频率等多维度建立分类标签体系，每个分类可独立配置展示样式与排序规则。

- **自动链接健康检查**：集成周期性链接可达性检测机制，自动标记失效或响应超时的外链，并在生成页面时以可视化状态标识予以警示。

- **静态站点生成引擎**：基于模板引擎将数据源编译为纯静态 HTML 文件，无需数据库或运行时环境，最大限度降低部署门槛与服务器资源消耗。

- **可扩展元数据体系**：每个链接条目支持自定义元数据字段，包括但不限于维护人、最后验证时间、备选地址、标签集合与备注说明，满足复杂管理需求。

- **全文检索支持**：生成过程中自动构建轻量级倒排索引，为导航页面提供标题与标签层面的即时搜索能力，提升资源定位速度。

- **版本化配置管理**：所有分类定义与链接数据均以纯文本文件形式存储，可纳入 Git 版本控制系统，便于追溯变更历史与团队协作。

- **响应式布局输出**：生成的导航页面默认适配桌面端与移动端浏览环境，确保在不同屏幕尺寸下均能获得一致的浏览体验。

## 应用场景

- **技术团队内部文档门户**：研发团队可将本系统作为内部技术文档的补充导航层，统一汇总常用的 API 参考、设计规范、运维手册等外部资源，减少新成员上手时的信息搜寻时间。

- **开源项目外部资源附录**：开源项目维护者可以在项目文档中集成由 Vanguard Link Aggregator 生成的资源列表页面，集中展示社区推荐的教程、工具链与相关生态项目，提升项目周边的信息组织质量。

- **个人知识库入口页面**：知识管理爱好者可将本系统作为个人知识库的起始页，分类收藏高频访问的技术博客、在线工具与学术文献站点，配合定期健康检查功能保持收藏夹的长期有效性。

- **技术活动或课程资料索引**：技术会议、培训课程或线上分享活动的主办方可利用本系统快速搭建课程资料导航页，将课件链接、实验环境地址、参考文献等集中呈现给参与者，无需依赖复杂的内容管理系统。

## 快速开始

以下操作指南假设您已具备基本的 Git 与 Node.js 使用经验。请按照顺序执行命令以完成项目的获取、依赖安装与初次生成。

```bash
# 从远程仓库克隆项目源码
git clone https://github.com/vanguard-labs/link-aggregator.git
cd link-aggregator

# 安装项目运行时依赖
npm install

# 执行构建命令生成静态站点
npm run build
```

执行完上述步骤后，生成的静态页面文件将输出至 `dist` 目录。您可以直接将该目录下的全部内容部署至任何静态 Web 服务器，或使用 `npm run serve` 命令启动本地预览服务。

## 项目结构

项目目录遵循模块化组织原则，核心源码与配置数据严格分离，便于维护与扩展。以下 ASCII 树形图展示了顶层目录结构及其职责说明。

```
link-aggregator/
├── config/                        # 全局配置目录
│   ├── categories.yaml            # 分类层级定义与展示排序配置
│   └── settings.json              # 站点元数据、构建参数与国际化选项
├── src/                           # 核心源码目录
│   ├── core/                      # 构建引擎核心模块
│   │   ├── parser.js              # 解析 YAML 数据源并转换为内部数据模型
│   │   ├── validator.js           # 校验链接格式与必需字段完整性
│   │   └── generator.js           # 协调模板渲染与静态文件输出流程
│   ├── templates/                 # 页面模板目录
│   │   ├── layout.hbs             # 基础 HTML 骨架与全局样式引用
│   │   ├── index.hbs              # 导航首页模板，渲染分类与链接列表
│   │   └── partials/              # 可复用的模板片段（导航栏、页脚、搜索框）
│   ├── assets/                    # 静态资源目录
│   │   ├── styles/                # SCSS 样式源文件与编译后的 CSS
│   │   └── scripts/               # 前端交互脚本（搜索、链接状态过滤）
│   └── utils/                     # 通用工具函数集合
│       ├── fetch-checker.js       # 实现 HTTP 链接可达性检测逻辑
│       └── index-builder.js       # 构建关键词倒排索引供前端检索使用
├── data/                          # 用户数据目录（核心维护位置）

│   │   ├── media.yaml             # 媒体类资源链接条目

├── .gitignore                     # Git 忽略文件规则
├── package.json                   # 项目依赖清单与脚本命令定义
├── README.md                      # 项目说明文档（即本文档）
└── LICENSE                        # MIT 许可证全文
```

## 常见问题

**问：健康检查功能是否会因为访问频率过高而被目标站点封禁？**

本项目内置的健康检查模块采用单线程顺序执行方式，且每次请求之间强制插入至少 200 毫秒的延迟间隔。该设计旨在将请求速率控制在普通人工浏览水平之下，最大程度降低对目标服务器的冲击。此外，检查模块遵循 HTTP 缓存头与重定向标准，不会对目标站点产生异常流量负担。若目标站点明确通过 `robots.txt` 禁止爬虫，建议在配置文件中为该域名关闭自动检查。

**问：生成大量链接条目后，构建速度是否会显著下降？**

本项目的构建性能主要受限于 I/O 操作（读取数据文件与写入生成的 HTML 文件）。对于数千条级别的链接数据，构建过程通常在 5 秒内完成。内部数据模型采用高效的内存映射与索引结构，分类归并与模板渲染的算法复杂度被控制在 O(n log n) 以内。若链接条目超过一万条，建议启用配置中的增量构建模式，仅重新生成发生变化的部分页面。

**问：如何在不重新构建整个站点的情况下更新单个链接的状态？**

链接状态（包括可达性与最近验证时间）被设计为独立于主数据文件存储。您可以通过运行 `npm run check:single -- --url <目标链接>` 命令单独触发某个链接的健康检查，该操作仅更新本地缓存的状态文件，无需触发全量构建。若您需要批量更新状态，建议使用 `npm run check:all` 命令，并在完成后执行 `npm run build -- --fast` 以利用缓存数据快速生成更新的页面版本。
