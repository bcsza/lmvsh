# HyperLink Nexus

HyperLink Nexus 是一个面向互联网基础设施运维、信息安全审计及网络拓扑测绘领域的开源外链资源聚合与健康度监测平台。项目定位于为网络工程师、安全分析师及站点可靠性工程师提供一套高可用的外部链接引用规范、状态追踪及元数据管理工具集，解决大规模分布式系统中外部依赖不可控、链接腐化及引用关系混乱等核心痛点。

本项目不提供具体业务功能，而是作为技术决策的知识索引层，将分散于各类技术博客、官方文档、社区讨论及私有归档中的高价值 URL 进行结构化整理，并辅以自动化可用性检查脚本，确保引用资源的长期有效性与可追溯性。

## 功能概览

- **资源分类索引引擎**：支持按行业领域、文件类型、更新频率等多维度对收录链接进行自动标签化与分类存储。
- **链接健康度探测模块**：内置基于 HTTP/HTTPS 状态码、响应时长及 TLS 证书有效期的周期性巡检任务，输出 JSON 格式的可用性报告。
- **元数据提取与缓存**：对每个收录 URL 自动抓取页面标题、描述关键词及最后修改时间，降低重复访问对源站造成的压力。
- **变更追踪与差分对比**：记录链接内容的历史快照哈希值，当目标页面发生显著变动时通过日志系统输出告警。
- **批量导入与导出适配器**：支持 CSV、Plaintext 及 Markdown 表格格式的链接批量入库与导出，便于与其他 CMDB 或监控系统对接。
- **权限分层查看机制**：内置访客、贡献者、管理员三级视图，控制敏感链接（如内网运维面板或测试环境入口）的可见范围。
- **自定义标签与注释系统**：允许用户为每个链接添加技术栈标签（如 Kubernetes、Terraform、Nginx）及人工运维备注。
- **RESTful API 查询接口**：提供基于时间范围、状态码、标签组合的复杂查询能力，返回结构化数据供下游自动化流水线消费。

## 应用场景

- **企业技术选型调研**：技术委员会在评估 API 网关或服务网格方案时，可使用本仓库预置的官方文档、性能对比文章及社区踩坑记录集合，快速建立信息全景视图，减少重复搜索成本。
- **网络边界安全策略验证**：安全团队可定期拉取本仓库中标记为外部 CDN、第三方认证服务及日志接收端点的链接列表，验证防火墙规则与 DNS 解析策略的有效性，及时发现被篡改或失效的依赖端点。
- **文档站点死链修复**：大型开源项目或企业内部 Wiki 维护者可利用本项目的健康度探测输出，定位文档中嵌入的失效引用链接，批量生成替换建议，提升用户体验。
- **渗透测试信息收集**：红队人员在开展外部攻击面梳理时，可依据本仓库分类好的技术论坛、代码托管平台及历史漏洞公告链接，快速定位目标系统可能涉及的第三方组件与已知薄弱环节。

## 快速开始

以下操作以 Linux/macOS 环境为例，演示从克隆到运行基础探测任务的完整流程。

```bash
# 克隆仓库到本地
git clone https://github.com/nexus-open/hyperlink-nexus.git
cd hyperlink-nexus

# 安装依赖（建议使用 Python 3.10 以上版本）
pip install -r requirements.txt

# 运行示例探测任务（默认使用 resources/sample_urls.txt 作为输入）
python probe.py --input resources/urls_24_48.txt --output reports/status_24_48.json --timeout 5 --retry 2
```

## 项目结构

```text
hyperlink-nexus/
├── probe.py                    # 主入口脚本，解析命令行参数并调度探测任务
├── requirements.txt            # Python 依赖锁定文件，固定版本保证环境一致性
├── config/                     # 配置目录，存放 YAML 格式的全局设定
│   ├── default.yaml            # 默认超时、重试、并发数等基础参数
│   └── tags_mapping.yaml       # 标签到正则表达式的映射规则，用于自动分类
├── core/                       # 核心功能模块包
│   ├── __init__.py
│   ├── fetcher.py              # 异步 HTTP 请求封装，含连接池与 SSL 上下文
│   ├── parser.py               # HTML 元数据解析，提取 title 与 meta 信息
│   ├── checker.py              # 状态码与证书有效期综合检查逻辑
│   └── cache.py                # 基于磁盘的持久化缓存，减少重复网络开销
├── resources/                  # 静态资源与输入文件存放目录
│   ├── sample_urls.txt         # 示例链接列表，供初次测试使用
│   └── urls_24_48.txt          # 当前批次原始 URL 清单（即本节所列链接）
├── reports/                    # 探测结果输出目录（自动生成，不入仓）
│   └── .gitkeep                # 保持目录存在
├── tests/                      # 单元测试与集成测试脚本
│   ├── test_fetcher.py         # 模拟异常响应、超时场景的测试用例
│   └── test_parser.py          # 针对不同 HTML 结构的解析鲁棒性测试
└── docs/                       # 完整文档根目录
    ├── user_guide.md
    ├── administration.md
    ├── development.md
    └── design_philosophy.md
```

## 贡献指南

1. 在 GitHub Issues 中查找或新建与链接分类、探测准确性或性能优化相关的问题，等待维护者标注 `ready-to-take` 标签后再行开发，避免重复劳动。
2. 克隆仓库并创建独立的特性分支，分支命名遵循 `feature/功能简述` 或 `fix/问题简述` 格式，禁止在主分支直接提交。
3. 新增或修改链接资源时，需同步更新 `resources/` 下对应批次的 `.txt` 文件，并在提交信息中附带变更说明，格式为 `[batch-24/48] add/remove/update url`。
4. 所有代码变更需通过 `pytest` 单元测试套件，并确保新增功能具备不少于 80% 的行覆盖率，测试报告将自动在 CI 流水线中生成。
5. 提交 Pull Request 时，需填写模板中的检查项（包括是否更新文档、是否兼容现有配置格式、是否进行过本地端到端探测验证），至少获得一名维护者的批准后方可合并。

## 常见问题

**问：探测脚本频繁被目标站点拒绝连接，如何调整行为以避免被误判为攻击流量？**

答：您可以在配置文件中调整 `request_interval` 参数，单位为秒，用于控制每次请求之间的最小间隔。同时建议启用 `random_user_agent` 选项以轮换客户端标识。对于特定敏感域名，可在 `resources/whitelist.yaml` 中单独设置更低的并发度与更长的超时阈值。

**问：部分 URL 返回 403 或 429 状态码，但通过浏览器手动访问正常，如何解决？**

答：这是由于目标站点启用了反机器人策略，例如校验 Cookie 或特定的请求头顺序。您可以在 `core/fetcher.py` 中取消注释 `custom_headers` 部分，并填入从浏览器开发者工具中获取的 `User-Agent`、`Accept-Language` 及 `Referer` 值。项目默认提供的精简请求头仅用于最小化指纹暴露，并非适用于所有站点。

**问：如何将本项目的探测结果集成到现有的 Prometheus + Grafana 监控体系中？**

答：`reports/` 目录下生成的 JSON 文件包含每个 URL 的状态码、响应时间（毫秒）及证书剩余天数。您可以使用 `prometheus` 目录下的 `exporter.py` 辅助脚本，该脚本将关键指标转换为 Prometheus 指标格式并暴露于 `:8000/metrics` 端点，随后在 Prometheus 配置中添加该任务即可。
