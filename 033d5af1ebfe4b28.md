# LinkVault

一个轻量级、高可扩展的技术资源外链聚合与导航系统。LinkVault 定位于解决开发者在日常工作中检索优质技术资源、影视工具站点、数据源地址时面临的碎片化问题，提供一套集中化、标签化、状态可监控的统一资源管理门户。项目本身不存储任何第三方内容，仅作为可信外链的索引与路由层，适用于个人知识管理、团队技术文档站点、社区导航页等场景。LinkVault 以静态站点形式部署，资源数据基于 YAML 配置文件驱动，支持全量检索、分类筛选、可用性拨测与访问统计，适合需要自主可控、轻运维、高定制性的技术团队或个人站长使用。

## 功能概览

- **分类导航体系**：按影视、数据、工具、开发库等一级分类组织资源条目，每个资源支持多级标签与业务域标注，便于快速定位。

- **外链可用性监控**：内置基于 HTTP HEAD 请求的异步拨测任务，可配置检查间隔与超时阈值，异常状态实时标记并记录历史可用率。

- **全文检索与过滤**：支持资源名称、描述、标签、域名的模糊搜索，同时提供分类、状态、协议类型的多维度筛选器。

- **访问统计与点击追踪**：每条外链记录点击次数、最后访问时间，支持简单热度排序，帮助识别高频使用资源。

- **数据配置热加载**：资源清单以 YAML 格式存放于 `./data/links.yaml`，修改后无需重启服务，通过 Webhook 或定时任务触发重新加载。

- **响应式暗色主题**：前端界面同时适配桌面与移动设备，内置亮色/暗色主题切换，减少夜间使用时的视觉疲劳。

- **开放 API 端点**：提供 `/api/v1/links`、`/api/v1/categories`、`/api/v1/status` 等 JSON 接口，方便与其他监控系统或内部工具集成。

- **导入导出能力**：支持从 CSV 批量导入资源条目，导出为 JSON 或 Markdown 列表格式，便于备份或迁移至其他导航工具。

## 应用场景

- **个人技术收藏夹的规范化管理**：开发者可将长期积累的零散书签按分类、标签结构化录入 LinkVault，配合全文检索快速找回，替代浏览器自带的扁平化书签系统。

- **团队内部基础设施导航页**：DevOps 团队可将跳板机、监控面板、日志系统、镜像仓库等内部地址统一录入，设置访问权限（通过反向代理层），作为团队首页使用。

- **开源社区资源聚合站**：技术社区或自媒体作者可以基于 LinkVault 搭建面向特定领域（如前端工具链、AI 模型库、影视转码工具）的外链导航站，降低新人寻找入门资源的门槛。

- **影视资源索引站**：针对用户提供的多个影视类域名（如 bearfilm.xyz、dsxfilm.skin 等），可作为统一入口，按类型、地区、更新频率组织，并辅以可用性拨测，帮助用户跳过已失效的站点。

## 快速开始

以下操作以 Linux/macOS 环境为例，假定已安装 Git、Node.js 18+ 与 pnpm。

```bash
# 克隆仓库
git clone https://github.com/your-org/linkvault.git
cd linkvault

# 安装依赖（使用 pnpm）
pnpm install

# 复制示例配置文件并编辑资源列表
cp ./config/links.example.yaml ./data/links.yaml
vim ./data/links.yaml

# 启动开发服务器（默认监听 3000 端口）
pnpm run dev
```

生产环境部署建议执行 `pnpm run build` 生成静态文件，随后通过 Nginx 或 Caddy 托管 `./dist` 目录。若需启用可用性拨测，请额外配置 `./config/monitor.json` 中的检查间隔与超时参数，并运行 `pnpm run monitor` 作为后台守护进程。

## 贡献指南

1. 在 GitHub 上 Fork 本仓库，并克隆到本地开发环境。建议在 `develop` 分支上切出新的特性分支，命名格式为 `feature/功能简述` 或 `fix/问题描述`。

2. 本地完成代码或文档修改后，请确保通过所有单元测试与 ESLint 检查（执行 `pnpm run lint` 与 `pnpm run test`）。若新增资源条目，请同步更新 `./data/links.yaml` 并运行校验脚本 `pnpm run validate`。

3. 提交信息请遵循 Conventional Commits 规范（如 `feat: add category filter`、`docs: update deployment guide`），便于自动生成变更日志。

4. 推送分支后，在 GitHub 上发起 Pull Request 至 `main` 分支。PR 描述中请说明改动目的、影响范围以及是否涉及破坏性变更。CI 通过且至少一位维护者 Approve 后方可合并。

5. 若发现资源链接失效或配置格式问题，欢迎提交 Issue，并按模板填写当前行为、期望行为及复现步骤。对于紧急链接修复，可直接提交 PR 并标记为 `hotfix`。

## 常见问题

**Q：LinkVault 是否存储或缓存第三方站点的内容？**

A：不存储。项目仅保留外链地址、标题、描述、标签等元数据，所有内容始终指向原始 URL。拨测仅检查 HTTP 状态码与响应头，不下载页面正文。用户访问外链时，流量直接由浏览器发起，不经过 LinkVault 服务端代理。

**Q：links.yaml 文件更新后需要重启服务吗？**

A：默认情况下，开发模式（`pnpm run dev`）会监听配置文件变化并自动热重载。生产环境如需热加载，需启用 `config/app.json` 中的 `watchLinks` 选项，或通过调用 `/api/admin/reload` 接口（需配置管理员令牌）触发重新加载。若不开启热加载，则需执行 `pnpm run build` 重新生成静态文件并重启 Web 服务器。

**Q：拨测任务会影响资源站点的正常访问吗？**

A：拨测采用 HEAD 请求，仅获取响应头信息，不下载 body，单次请求开销极小。默认超时时间为 3 秒，重试次数为 1，间隔为 30 分钟，对绝大多数站点的负载影响可忽略。若需调低检查频率，可修改 `config/monitor.json` 中的 `intervalMinutes` 字段。
