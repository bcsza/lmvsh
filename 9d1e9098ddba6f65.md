# Project Atlas

An open-source, community-driven index of digital media resources and niche streaming platforms. Project Atlas serves as a structured, human-maintained catalog of specialized video content sites, domain metadata, and accessibility endpoints. It targets technical users, digital archivists, and media researchers who require a clean, queryable reference for discovering and categorizing lesser-known streaming and video-sharing services across non-standard top-level domains.

The project does not host, proxy, or redistribute any media content. It provides a verified list of resource locations, platform tags, status checking scripts, and a lightweight API for integration into custom dashboards or monitoring tools. Project Atlas is designed to reduce the discovery friction for curators and developers who need to maintain their own secondary indexes of external media sources without relying on commercial search engines or centralized recommendation algorithms.

## 功能概览

- **Curated Domain Registry** – Maintains a manually validated list of active streaming and video-sharing domains with last-seen timestamps and content type tags.

- **Automated Health Check Scripts** – Includes a Python-based availability tester that performs HTTP HEAD and GET probes against each listed domain, recording response times and status codes.

- **Categorized Tagging System** – Each resource is tagged with primary categories such as `drama`, `animation`, `variety`, `documentary`, and `regional` to enable filtered lookups.

- **JSON Export Pipeline** – Provides scheduled export of the registry to JSON and CSV formats for easy ingestion into external analytics or monitoring systems.

- **Lightweight REST Query Interface** – A simple Flask-based endpoint that accepts tag filters, status filters, and free-text searches over the registry fields.

- **Metadata Enrichment Module** – Optional integration with public WHOIS and DNS lookup services to enrich each record with registrar info, nameserver details, and creation dates.

- **Configuration Override System** – Allows operators to set custom timeouts, retry policies, and user-agent strings for the health checker through a single YAML configuration file.

- **Logging and Alert Hook** – Built-in file-based logging with an optional webhook forwarder (Discord/Slack compatible) for notifying administrators of domain status changes.

## 应用场景

- **Personal Media Aggregator Dashboard** – A developer can deploy the Atlas health checker as a cron job and feed the JSON output into a simple web dashboard to monitor which video sites are currently responsive, enabling quick manual access without visiting each URL individually.

- **Research Archive Maintenance** – An academic researcher studying regional streaming trends can use the tagged registry to periodically sample content availability and collect uptime statistics across different TLDs, generating data for longitudinal analysis of platform stability.

- **Internal Enterprise Allowlist Curation** – An IT team managing a corporate media library can integrate the Atlas export module to generate a daily allowlist of approved video domains, ensuring that only verified and tagged resources are accessible through internal proxy gateways.

- **DevOps Monitoring Pipeline** – A site reliability engineer can attach the health check script to a Prometheus exporter, creating alerts when certain high-priority domains fail consecutive checks, thus proactively detecting service disruptions before user complaints arise.

- **Open Source Data Feed** – A data journalist can subscribe to the weekly JSON dump from the public mirror and cross-reference the Atlas registry with other public datasets to identify correlations between domain registration patterns and content release schedules.

## 快速开始

Clone the repository, install dependencies, and run the initial health check on the default registry.

```bash
git clone https://github.com/atlas-project/atlas-index.git
cd atlas-index
pip install -r requirements.txt
python health_check.py --registry data/registry.json --output reports/status.json
python run_api.py --port 8080
```

To schedule regular updates, add the health check command to your crontab:

```bash
0 */6 * * * cd /opt/atlas-index && python health_check.py --registry data/registry.json --output reports/status_$(date +\%Y\%m\%d-\%H).json
```

## 贡献指南

We welcome contributions that improve the reliability, coverage, or usability of the registry and its tooling. Please follow the steps below to contribute effectively.

1. Fork the repository and create a feature branch from `main` with a descriptive name, such as `feature/add-enrichment-module` or `fix/checker-timeout-handling`.

2. If you are adding or modifying domain entries, ensure you update the registry JSON file and run the validation script (`python src/cli/registry_tool.py validate`) to confirm schema compliance. Include a brief comment in the pull request explaining the source or rationale for each added domain.

3. For code changes, write or update relevant unit tests under the `tests/unit/` directory and ensure all existing tests pass by running `pytest` locally. Use `black` and `mypy` to format and type-check your code before committing.

4. Update the documentation files under `docs/` if your changes affect user-facing behavior, configuration options, or API responses. Include examples where applicable.

5. Submit a pull request with a clear title and a detailed description of the changes, referencing any related issues. Pull requests that lack test coverage or documentation updates will be flagged for revision before merging.

## 常见问题

**Q: The health check script reports a timeout for some domains that are still accessible via browser. Why does this happen?**

A: The health checker uses a default timeout of 10 seconds and follows standard HTTP redirects. Some media sites may employ anti-bot mechanisms, such as JavaScript-based challenges or delayed responses that exceed the timeout. You can increase the timeout value by editing the `http_timeout` field in your `config/custom.yaml` override file. Additionally, check your network environment, as corporate firewalls or VPNs may interfere with outbound probes.

**Q: How frequently is the public registry updated? Can I host my own private copy?**

A: The public registry JSON is updated weekly through automated scheduled runs. For private deployments, you can host your own copy by cloning the repository and running the health checker against your own curated list of domains. The `data/registry.json` file is designed to be easily replaceable or extended with your own entries, provided you maintain the expected schema format.

**Q: Is there any authentication required for the REST API?**

A: By default, the API runs without authentication to simplify local and internal network deployments. For production environments exposed to the internet, we recommend placing the API behind a reverse proxy with basic authentication or API key validation. An example NGINX configuration with auth_basic is provided in the `docs/deployment/` directory.
