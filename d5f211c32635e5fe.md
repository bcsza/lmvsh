# ApertureNav

ApertureNav 是一个面向技术实践者与信息研究人员的轻量级外链聚合与导航系统。项目本身不存储任何实质内容，仅提供结构化、可维护的链接索引与分类视图，帮助用户在信息过载的环境中快速定位高价值技术资源、社区站点与动态更新源。项目定位为“技术路标而非内容仓库”，适用于个人部署、团队内部分享以及公开实例运营。

目标用户包括运维工程师、全栈开发者、安全研究人员以及技术内容策展人。项目解决的核心痛点是：优质技术站点分散、个人书签难以协作、新人缺乏可信入口、以及项目文档中的外部引用缺乏统一管理视图。通过 ApertureNav，用户可以获得一个可编程、可版本化、可审计的外部资源索引层。

## 功能概览

- **多级分类索引**：支持按技术领域、站点性质、更新频率等维度对链接进行标签化分组，每个分组可独立配置可见性与描述信息。

- **裸链接直出模式**：所有外链以原始字符串形式呈现，不附加追踪参数、跳转中间页或重写逻辑，确保访问路径透明且可预期。

- **批量导入与校验**：支持通过 YAML 或 JSON 格式批量导入链接列表，系统自动执行可达性检查与协议一致性校验，并生成异常报告。

- **静态站点生成**：基于配置数据输出完整的静态 HTML 目录页面，可直接托管于对象存储、CDN 或轻量级 Web 服务器，无需运行时数据库。

- **变更审计日志**：记录每次链接增删改的操作上下文，支持回滚至任意历史版本，满足团队协作与合规追溯需求。

- **自定义元数据扩展**：每条链接允许附加自定义键值对，例如维护人、过期时间、替代入口、访问频率建议等，便于实现上层自动化流程。

- **响应式搜索与过滤**：内置模糊搜索与正则表达式过滤能力，支持实时筛选当前视图下的链接条目，无需刷新页面。

## 应用场景

1. **技术团队内部知识入口**  
   开发团队可将 ApertureNav 部署为内部首页，集中放置 CI/CD 系统、日志平台、监控面板、镜像仓库及内部文档站点的访问入口，新成员入职时可快速了解团队常用工具链。

2. **开源项目外部依赖索引**  
   开源项目维护者可在项目文档中引用 ApertureNav 实例，集中列出上游依赖的官方文档、社区论坛、参考实现及备选镜像源，避免在 README 中堆砌冗长链接列表。

3. **安全研究情报聚合**  
   安全研究人员可利用 ApertureNav 分类管理威胁情报源、漏洞数据库、沙箱环境、样本分析平台及行业报告站点，结合元数据标记数据新鲜度与可信等级。

4. **个人书签的版本化管理**  
   个人开发者可将浏览器书签迁移至 ApertureNav 的配置文件，通过 Git 进行版本管理，实现多设备同步与变更历史追溯，同时降低浏览器书签丢失或混乱的风险。

5. **技术社区资源导航站**  
   技术社区运营方可使用 ApertureNav 搭建公开导航站点，按语言、框架、领域等维度整理优质外链，降低社区成员的信息检索成本，并支持社区提交新链接的审核流程。

## 快速开始

以下命令用于在 Linux 或 macOS 环境下完成项目克隆、依赖安装及服务启动。Windows 用户建议通过 WSL2 或 Git Bash 执行。

```bash
git clone https://github.com/aperture-nav/aperture-nav.git
cd aperture-nav
npm install
npm run build && npm start
```

执行完毕后，默认在 127.0.0.1:3000 启动 Web 服务。如需修改端口或绑定地址，可编辑 config/server.yaml 中的 listen 字段。生产环境建议使用 pm2 或 systemd 进行进程管理。

## 贡献指南

1. 在 GitHub 上 fork 本仓库，并 clone 至本地开发环境。建议在 dev 分支上进行修改，避免直接操作 main 分支。

2. 新增或修改链接数据时，请编辑 config/links/ 下的对应 YAML 文件，遵循既定格式规范（每条链接包含 url、title、description 及可选的 tags）。提交前运行 npm run validate 校验配置完整性。

3. 若新增分类或调整结构，需同步更新 config/categories.yaml 以及文档中的相关描述，确保导航层次与文档说明保持一致。

4. 提交代码时请遵守 Conventional Commits 规范，提交信息需明确说明变更类型（feat、fix、docs、chore 等）及影响范围。提交前务必通过 npm test 全部测试用例。

5. 欢迎提交文档改进、翻译或示例配置。对于重大功能变更，建议先创建 Issue 进行讨论，再提交 Pull Request，并附上充分的测试覆盖与使用示例。

## 常见问题

**问：为什么项目不直接存储或代理外部资源内容？**

答：ApertureNav 的设计目标仅为“索引”而非“存储”。不存储内容可规避版权与合规风险，同时保持项目轻量化，便于快速部署和迁移。用户始终直接访问原始资源地址，项目不介入内容传输路径。

**问：如何确保链接列表的长期有效性？**

答：项目提供可选的链接检查脚本（npm run check-links），可定期运行并生成失效链接报告。建议用户结合 CI 调度（如每周执行）自动生成检查结果，并通知相关维护人处理。但项目本身不强制自动移除失效链接，由用户根据策略决定处理方式。

**问：能否将 ApertureNav 与现有认证系统（如 LDAP、OAuth）集成？**

答：ApertureNav 本身不提供用户认证模块，但可通过反向代理层（如 nginx、caddy 或 oauth2-proxy）实现访问控制。静态生成模式下，也可将输出目录部署至支持基础认证的 Web 服务器。项目文档的运维章节提供了常见网关配置示例。
